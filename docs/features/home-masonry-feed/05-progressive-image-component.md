# 首页瀑布流步骤五：ProgressiveImage 组件设计

## 目的

本步骤用于实现首页瀑布流卡片中的渐进式图片组件 `ProgressiveImage`，为首页封面图提供稳定的首屏占位、灰骨架、LQIP 占位图、原图懒加载和淡入过渡能力。

本步骤的目标是：

- 将首页卡片封面的图片显示逻辑从卡片组件中抽离，形成可复用的独立组件。
- 在图片尚未就绪时，先稳定占位，避免瀑布流布局抖动。
- 在图片进入视口附近时，按三阶段流程展示：
  - 灰骨架
  - LQIP 占位图
  - 原图
- 在原图真正可以安全绘制后再切换显示，减少闪烁和跳帧。
- 保持组件职责单一，只负责展示，不负责业务查询或图片元数据生成。
- 为后续首页瀑布流卡片、Explore 类似卡片，甚至未来更多图片列表提供复用基础。

本步骤不负责：

- 首页分页查询
- 图片元数据生成
- 创建 / 编辑帖子写入元数据
- 瀑布流分列逻辑

## 验收标准

本步骤完成后，应满足以下验收标准：

- 新增 `ProgressiveImage` 组件，例如：
  - `app/features/post/components/ProgressiveImage.tsx`
- 组件输入至少包括：
  - `src`
  - `alt`
  - `aspectRatioBucket`
  - `placeholder`
- 组件首次渲染时，会先按 `aspectRatioBucket` 预留稳定布局空间。
- 首次渲染默认先显示灰骨架，而不是直接显示原图。
- 组件内部通过 `IntersectionObserver` 判断何时进入加载流程。
- 当组件接近视口时：
  - 如果有 `placeholder`，切换显示 LQIP
  - 同时启动原图加载
- 原图通过 `Image.decode()` 确认解码完成后再显示。
- 原图展示时有淡入过渡，placeholder 和灰骨架逐步退出。
- 原图 `<img>` 保留：
  - `loading="lazy"`
  - `decoding="async"`
- 当 `placeholder` 缺失时，组件仍能从灰骨架平滑过渡到原图。
- 当原图加载或解码失败时，组件不会无限卡在 pending 状态，应有明确降级行为。
- 组件不直接调用 TanStack Query，也不直接访问 Appwrite。

## 改了什么，改在哪里

### 一、新增 `ProgressiveImage` 组件

改动位置：

- `app/features/post/components/ProgressiveImage.tsx`

这是一个纯展示组件，建议职责限定为：

- 接收已准备好的图片数据
- 管理图片显示状态
- 渲染占位与原图层
- 处理接近视口时的加载时机
- 处理解码完成后的切换

不应承担的职责：

- 拉取帖子数据
- 生成 LQIP
- 写入图片元数据
- 处理上传

### 二、定义组件输入契约

建议输入：

- `src`
  - 原图 URL
  - 来源：首页 feed 查询结果中的 `imageUrl`
- `alt`
  - 替代文本
  - 来源：父组件根据 `caption`、作者名等组装
- `aspectRatioBucket`
  - 比例桶
  - 来源：首页 feed 查询结果中的 `aspectRatioBucket`
  - 旧数据会在 mapper 中默认降级到 `3:4`
- `placeholder`
  - LQIP data URL
  - 来源：首页 feed 查询结果中的 `imagePlaceholder`
  - 可为 `null`

可选扩展输入可后续再加，例如：

- `className`
- `sizes`
- `objectFit`
- `onLoad`

但第一版建议保持最小输入集。

### 三、组件内部状态设计

建议内部至少有以下状态：

- `idle`
  - 初始状态
  - 仅显示灰骨架
- `placeholder`
  - 已进入视口附近
  - 显示 LQIP
  - shimmer 可生效
- `full`
  - 原图已加载并 decode 完成
  - 原图淡入显示
- `error`
  - 原图加载或 decode 失败
  - 保留灰骨架或错误降级视图

状态职责：

- `idle` 负责首帧稳定性
- `placeholder` 负责中间过渡
- `full` 负责最终展示
- `error` 负责可恢复降级，避免无限 loading

### 四、首次渲染行为

首次渲染行为设计如下：

1. 组件拿到 `aspectRatioBucket`
2. 先渲染一个固定比例的图片容器
3. 容器里先显示灰骨架
4. 此时不立即把原图强行显示出来
5. 组件 mount 后开始观察元素是否接近视口
6. 当进入触发范围后：
  - 若有 `placeholder`，进入 `placeholder` 状态
  - 同时开始原图加载与解码
7. 原图 decode 成功后进入 `full` 状态，执行淡入

这样做的目的：

- 先稳定布局
- 再进入加载流程
- 让视觉更接近小红书首页“文字先出、图片区先灰块、再逐渐显图”的体验

### 五、`IntersectionObserver` 的使用方式

组件内部使用 `IntersectionObserver`，不是为了“取数据”，而是为了控制图片状态机的启动时机。

它的职责：

- 判断图片容器是否已接近视口
- 决定何时从 `idle` 进入 `placeholder` / 原图加载流程

建议行为：

- 使用 `rootMargin` 让图片在进入视口前提前加载，例如：
  - `200px`
  - `300px`
- 一旦进入观察范围并启动加载，可取消观察，避免重复触发

这样设计的原因：

- 让图片在真正出现在屏幕前就有机会准备好
- 避免在首页长列表里一次性加载过多离屏图片

### 六、`Image.decode()` 的使用方式

组件不应直接依赖 `<img onLoad>` 作为切换到原图的唯一信号。

建议流程：

1. 使用 `new Image()`
2. 设置 `img.src = src`
3. 等待图片加载
4. 调用 `await img.decode()`
5. 成功后再让真实原图层进入可见状态

这样做的原因：

- `load` 表示资源已到达，但并不总是意味着图片已经准备好平滑绘制
- `decode()` 可以让原图在显示前先完成解码
- 在瀑布流中，这能降低闪烁和切换卡顿的概率

### 七、保留原生浏览器兜底属性

原图 `<img>` 仍建议保留：

- `loading="lazy"`
- `decoding="async"`

作用分别是：

- `loading="lazy"`
  - 告诉浏览器该图片适合离屏延迟加载
- `decoding="async"`
  - 告诉浏览器图片解码可以异步进行，降低阻塞渲染的概率

这两个属性不是为了替代：

- `IntersectionObserver`
- `Image.decode()`

而是作为浏览器层的兜底提示，与组件内部状态机形成互补。

### 八、与首页 query 的关系

本组件不发起 query。

它依赖首页分页数据层已经查好以下字段：

- `imageUrl`
- `imagePlaceholder`
- `aspectRatioBucket`

并由父级卡片组件传入：

- `src`
- `placeholder`
- `aspectRatioBucket`
- `alt`

因此，本组件的 Query 设计原则是：

- 自己不查数据
- 只消费父层传下来的已归一化字段

这样职责边界最清晰。

## 为什么选择这个方案

### 为什么需要单独的 `ProgressiveImage`

首页瀑布流卡片会大量重复使用图片显示逻辑，如果把以下逻辑写在卡片组件里：

- 比例占位
- 骨架
- placeholder
- 懒加载
- decode
- fade-in

会让卡片组件很快失去可读性，并且后续在其他列表页复用时重复实现。

将其独立为 `ProgressiveImage` 的好处：

- 职责清晰
- 更容易测试和调试
- 可复用
- 卡片组件只关心排版和信息内容，不关心图片状态机细节

### 为什么组件内部用 `IntersectionObserver`

因为首页是瀑布流，不是简单的静态页面图片。

如果只依赖浏览器原生懒加载：

- 组件无法精确控制“何时从灰骨架进入 placeholder”
- 也无法把占位图显示时机和原图加载流程明确绑定在一起

使用 `IntersectionObserver` 后，组件能更主动地控制自己的显示状态机。

### 为什么要用 `Image.decode()`

因为图片“下载完成”与“可以无闪烁显示”不是同一件事。

`Image.decode()` 可以让组件在原图真正可绘制时再切换状态，减少以下问题：

- 原图突兀闪现
- 切换时抖动
- 首次显示时的帧卡顿

这对图片密集型的瀑布流首页尤其重要。

### 为什么保留 `loading="lazy"` 和 `decoding="async"`

因为它们是浏览器层面的轻量提示，几乎没有额外复杂度，却能提供合理兜底：

- 即使自定义状态机出现边界问题，浏览器仍知道这些图片适合延迟加载
- 即使 `decode()` 路径因某些异常降级，浏览器仍可异步调度图片解码

这是一个“自定义控制 + 浏览器兜底”的组合，而不是二选一。

### 为什么首次渲染先显示灰骨架

这不是技术限制，而是产品体验选择。

原因：

- 灰骨架能提供更稳定的首帧感知
- 视觉上更接近你观察到的小红书首页
- 在瀑布流里避免首屏出现大量彩色模糊块带来的噪音

因此，第一帧先灰骨架，再进入 placeholder 和原图，是更符合当前目标体验的方案。

## 实现顺序与依赖关系

### 第一步：完成首页分页数据层

依赖：

- 依赖步骤四中已完成：
  - 首页分页 API / service / query
  - 首页专用 select
  - 首页专用 view model

原因：

- `ProgressiveImage` 所需字段必须已能稳定从首页 query 中获取

### 第二步：确定组件输入契约

操作：

- 明确 `src`、`alt`、`aspectRatioBucket`、`placeholder`
- 确定父组件如何传入这些字段

依赖：

- 依赖首页 mapper 已完成字段归一化

### 第三步：实现比例占位和灰骨架

操作：

- 根据 `aspectRatioBucket` 生成稳定布局容器
- 渲染默认骨架层

依赖：

- 无额外依赖，可先独立完成

### 第四步：实现 `IntersectionObserver`

操作：

- 在元素接近视口时切换到加载流程
- 使用合理 `rootMargin`
- 启动后及时取消观察

依赖：

- 依赖容器 ref 和基础状态机

### 第五步：实现原图加载与 `decode()`

操作：

- 创建 `Image()` 进行原图加载
- 调用 `decode()`
- 成功后进入 `full`
- 失败后进入 `error`

依赖：

- 依赖加载触发时机已建立

### 第六步：补齐过渡样式与降级逻辑

操作：

- placeholder 与原图的透明度切换
- shimmer 和骨架退场
- 无 placeholder / decode 失败时的降级逻辑

依赖：

- 依赖状态机已成型

### 第七步：接入首页卡片

操作：

- 在首页卡片组件中使用 `ProgressiveImage`

依赖：

- 依赖首页卡片组件和瀑布流组件进入实现阶段

## 关键风险及应对策略

### 风险：状态机复杂后容易出现卡死状态

问题：

- 如果 `idle`、`placeholder`、`full`、`error` 状态切换设计不清晰，组件可能卡在 placeholder 或永远不进入 full。

应对：

- 明确状态转移边界
- 避免一个状态承担多个语义
- 对 decode 失败和 placeholder 缺失都定义清晰降级路径

### 风险：快速滚动导致异步回调操作已卸载组件

问题：

- `IntersectionObserver`、`Image.decode()` 和异步加载都可能在组件卸载后继续回调。

应对：

- 使用 mounted 标记或等价保护
- 卸载时断开 observer
- 对异步结果落地前做有效性检查

### 风险：placeholder 缺失时显示体验退化

问题：

- 旧帖子或元数据生成失败的帖子可能没有 `imagePlaceholder`

应对：

- 将 placeholder 缺失视为正常降级路径
- 组件从灰骨架直接过渡到原图
- 不因缺少 placeholder 而影响首页整体可用性

### 风险：`decode()` 在部分场景失败或不稳定

问题：

- 浏览器实现或图片资源异常可能导致 `decode()` reject

应对：

- 捕获 `decode()` 异常
- 降级为普通图片加载完成后展示，或进入 `error`
- 不让组件无限等待

### 风险：比例占位与实际图片视觉不一致

问题：

- 由于使用的是比例桶而不是真实比例，个别图片可能在视觉上与真实尺寸存在轻微偏差

应对：

- 这是当前设计允许的近似策略
- 优先保证首页布局稳定性
- 后续如果需要更高还原度，可基于 `imageWidth` / `imageHeight` 做更细粒度比例计算

### 风险：同时使用 `IntersectionObserver`、`loading="lazy"`、`decode()` 造成理解成本提高

问题：

- 三者都和图片加载有关，后续维护者可能混淆职责。

应对：

- 在组件和文档中明确分工：
  - `IntersectionObserver`：控制状态机启动
  - `loading="lazy"`：浏览器层延迟加载提示
  - `decoding="async"`：浏览器层异步解码提示
  - `Image.decode()`：控制切换到原图的时机

## 预期结果

本步骤完成后，项目将拥有一个职责清晰、可复用、面向图片密集型 feed 的渐进式图片组件：

- 首页卡片能先稳定占位，再逐步显示图片内容
- 灰骨架、LQIP 和原图切换逻辑被集中封装
- 原图显示更平滑，减少闪烁和布局跳动
- 组件不与业务查询耦合，便于在未来其他卡片和列表中复用
